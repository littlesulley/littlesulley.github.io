An Introduction to Reservoir Sampling (with Pseudo Code)

Posted on 2023-05-28 Edited on 2023-05-29 In 数学 - 概率论 Disqus: Word count in article: 10k Reading time ≈ 9 mins.

Given a stream of data formalized as x1,x2,⋯ ,xn,⋯x_1, x_2, \cdots, x_n, \cdotsx1​,x2​,⋯,xn​,⋯, how will you randomly sample kkk of them when this stream ends. Apparently you cannot wait for the stream to end and cache all data due to a underlying prohibitively large data scale, neither can you sample new kkk pieces of data at each incoming step. Can we do this incrementally and guarantee that the sampled data are fully uniform? Yes, this leads us to the algorithm of Reservoir Samping.

Problem Definition

Let me first give a formal definition to our problem:

Let {xn}\{x_n\}{xn​} be a stream of data records where its length is unknown in advance. When the data stream finishes, return a set of kkk data records uniformly sampled from {xn}\{x_n\}{xn​}.

The most trivially solution is to store all data records and sample kkk indices from the index set {1,2,⋯ ,n}\{1, 2, \cdots, n\}{1,2,⋯,n}. But this is intractably storage-expensive as {xn}\{x_n\}{xn​} may exceed the maximum storaget limitation.

Can we procedurally take samples without caching each incoming data record? The answer is yes. I will introduce three common solutions in the following sections, including the famous Reservoir Sampling algorithm.

Priority Queue

Theory

When current incoming index i≤ki\le ki≤k, we just need to keep this data record xix_ixi​ and put it into the result set R←R∪{xi}\mathcal{R}\gets\mathcal{R}\cup\{x_i\}R←R∪{xi​}. When the index i>ki>ki>k, we must decide whether or not this data record should be stored into R\mathcal{R}R and which existing data record in R\mathcal{R}R should be replaced by xix_ixi​.

Because we want to uniformly sample data records, we can assign a unique uniform value uiu_iui​ sampled from U[0,1]\mathcal{U}[0,1]U[0,1] to each incoming data record and compare this value to those already stored kkk smallest values, denoted by P={uj1,uj2,⋯ ,ujk}\mathcal{P}=\{u_{j_1},u_{j_2},\cdots, u_{j_k}\}P={uj1​​,uj2​​,⋯,ujk​​}. If uiu_iui​ is smaller than the largest of P\mathcal{P}P, the newly sampled value uiu_iui​ will be inserted into P\mathcal{P}P, and the old largest value will be pruned out.

The theory behind is simple:

Given nnn random variables sampled from U[0,1]\mathcal{U}[0,1]U[0,1], the indices of the smallest kkk of them is a uniform sample of the kkk-subsets of {1,2,⋯ ,n}\{1, 2, \cdots, n\}{1,2,⋯,n}.

How can we efficiently insert a new value and maintain the largest one? Priority queue, or maximum heap serves our purpose well. By definition, we can maintain a priority queue P\mathcal{P}P of size kkk and at each incoming step we sample a value from U[0,1]\mathcal{U}[0,1]U[0,1]. If the sampled value is smaller than the largest in P\mathcal{P}P, we remove the largest and insert the new one as well as the associated data record, which can be done using a pair.

Complexity

The running time complexity can be divided into two parts (ignoring the first kkk data records): one for when the sampled value rrr is less than the current largest value in P\mathcal{P}P; the other for each incoming data record. For the latter, as each data record has be to read, the total time complexity is O(n)\mathcal{O}(n)O(n); for the former, when the sampled value rrr is less than the largest value in $\mathcal{P}, the whole priority queue needs to be re-arranged which makes it take an additional time complexity of O(log⁡(k))\mathcal{O}(\log(k))O(log(k)).

The expected running complexity is:

\begin{equation} \begin{aligned} &~~~~~n + \log(k)\sum_{i=k+1}^n\mathrm{Pr}(i~\text{is selected})\\ &=n+\log(k)\sum_{i=k+1}^n\frac{k}{i}\\ &=n+k\log(k)\log(n/k) + \mathcal{O}(1) \end{aligned} \end{equation}

The first equality holds because all records have the same chance to enter the priority queue and are uniformly sampled based on U[0,1]\mathcal{U}[0,1]U[0,1]. For the iii-th record (i>ki>ki>k), it has a chance of k/ik/ik/i to replace one value in the priority queue. So its probability being selected, or having a value less than the largest in the priority queue, is exactly k/ik/ik/i.

The second equality holds according to the formula ∑i=1n1i=log⁡(n)+O(1)\sum_{i=1}^n\frac{1}{i}=\log(n)+\mathcal{O}(1)∑i=1n​i1​=log(n)+O(1).

Therefore, the final expected running complexity is O(n+klog⁡(k)log⁡(n/k))\mathcal{O}(n+k\log(k)\log(n/k))O(n+klog(k)log(n/k)).

Reservoir Sampling

Theory

Can we avoid using any external data structure, like priority queue, to uniformly sample what we want? Yes! And it can be much easier.

It goes as follows:

  1. Cache the first kkk data records and store them into the result set R\mathcal{R}R.
  2. For the current incoming step i(i>k)i (i>k)i(i>k), sample an integer jjj from {1,2,⋯ ,k,⋯ ,i}\{1, 2, \cdots, k, \cdots, i\}{1,2,⋯,k,⋯,i}. If j∈{1,2,⋯ ,k}j\in \{1, 2, \cdots, k\}j∈{1,2,⋯,k}, set R[j]=S[i]\mathcal{R}[j]=\mathcal{S}[i]R[j]=S[i], otherwise we discard S[i]\mathcal{S}[i]S[i] and proceed.

That is, at each step we sample an index from the current index set. If the sampled integer is within the first kkk indices, we replace the corresponding stored record with current incoming record; otherwise we do nothing.

The core idea to prove it is to show at each step iii, the probability of each seen record x1,x2,⋯ ,xix_1, x_2, \cdots, x_ix1​,x2​,⋯,xi​ being selected is k/ik/ik/i. Assume this holds for iii and we will prove it for i+1i+1i+1.

For an already selected data record, say xjx_jxj​, its probability being selected at step iii is k/ik/ik/i by induction. At this turn (i+1i+1i+1), the probability of xjx_jxj​ still being selected is:

\begin{equation} \begin{aligned} \text{Pr}(x_j~\text{is still selected in next turn}) &= \frac{k}{i}\left(1-\frac{k}{i+1}\right)+\frac{k}{i}\frac{k}{i+1}\frac{k-1}{k}\\ &=\frac{k}{i}\frac{i}{i+1}\\ &=\frac{k}{i+1} \end{aligned} \end{equation}

This closes our proof. The first term means xi+1x_{i+1}xi+1​ is not selected and the second term means xi+1x_{i+1}xi+1​ is selected but jjj is not selected. Both cases lead to the remaining of xjx_jxj​.

This algorithm is called Reservoir Sampling.

*PS: I don’t know why it’s given this name, maybe because it stores kkk samples from a population of data records like a reservoir?

Complexity

Each incoming data record needs to be read from source streaming. Hence, the running complexity is O(n)\mathcal{O}(n)O(n).

Improved Reservoir Sampling

Theory

Can we further improve the vanilla reservoir sampling algorithm? You may ask, how can we achieve this if every record has to be read from streaming. Yes, the optimal complexity is O(n)\mathcal{O}(n)O(n) if each record is read. But what if we skip some records and straightly locate to the next selected record? Well, with this in mind, we can indeed improve the original reservoir sampling algorithm.

Going back to what we did in priority queue, we sample a value from the uniform distribution U[0,1]\mathcal{U}[0,1]U[0,1] for each data record and compare it to existing cached largest value. In other words, denote the current maximum value by www, the probability of selecting current data record xi+1x_{i+1}xi+1​ will be www, and the probability of not selecting it is 1−w1-w1−w. Extending it to lll steps further. The probability of selecting xi+lx_{i+l}xi+l​ is (1−w)l−1w(1-w)^{l-1}w(1−w)l−1w, or in the form of (1−w)l−1−(1−w)l(1-w)^{l-1}-(1-w)^{l}(1−w)l−1−(1−w)l. Accumulate lll from 111 to infinity, we have the following cumulative distribution function (CDF):

F(xi+l is selected)=∑j=1lPr(xi+j is selected)=1−(1−w)lF(x_{i+l}~\text{is selected})=\sum_{j=1}^l\text{Pr}(x_{i+j}~\text{is selected})=1-(1-w)^l F(xi+l​ is selected)=j=1∑l​Pr(xi+j​ is selected)=1−(1−w)l

With that being said, if xix_ixi​ is selected, we can sample a value r∼U[0,1]r\sim\mathcal{U}[0,1]r∼U[0,1] and set r=1−(1−w)lr=1-(1-w)^lr=1−(1−w)l. Then we successfully sample the next selection index l=floor(log⁡(r)log⁡(1−w))+1l=\text{floor}\left(\dfrac{\log(r)}{\log(1-w)}\right)+1l=floor(log(1−w)log(r)​)+1. Here we set r←1−rr\gets 1-rr←1−r since rrr is uniformly sampled.

Now that we obtain our next selection index i+li+li+l, what is www then? Apparently we cannot keep it as it is because the new sample wi+lw_{i+l}wi+l​ must have a smaller value than www. Well, actually wi+lw_{i+l}wi+l​ has the same distribution as max⁡{u1,⋯ ,uk}\max\{u_1,\cdots,u_k\}max{u1​,⋯,uk​} where all u1,⋯ ,uk∼U[0,w]u_1, \cdots, u_k\sim\mathcal{U}[0,w]u1​,⋯,uk​∼U[0,w] independenly. Here is the reason (without strict proof):

  • Each stored record has the same chance to be replaced by xi+lx_{i+l}xi+l​ as they are all randomly sampled.
  • And each stored record’s value uj(j=1,⋯ ,k)u_j (j=1, \cdots, k)uj​(j=1,⋯,k) is sampled from U[0,w]\mathcal{U}[0,w]U[0,w] since we only know the maximum www and the real value thus lies in a uniform distribution U[0,w]\mathcal{U}[0,w]U[0,w].
  • We only care about the maximum of them and treat the maximum as the new www, i.e., w=max⁡{u1,⋯ ,uk}w=\max\{u_1,\cdots,u_k\}w=max{u1​,⋯,uk​}.

Now, calculating new www reduces to the follow problem: given a uniformly and independently sampled value set {u1,⋯ ,uk}\{u_1,\cdots,u_k\}{u1​,⋯,uk​} from U[0,w]\mathcal{U}[0,w]U[0,w], how to sample their maximum max⁡{u1,⋯ ,uk}\max\{u_1,\cdots,u_k\}max{u1​,⋯,uk​}. This can be done by first sampling a random value r∼U[0,1]r\sim\mathcal{U}[0,1]r∼U[0,1] and then take w⋅r1/kw\cdot r^{1/k}w⋅r1/k.

Here is the proof:

Denote the maximum by u=max⁡{u1,⋯ ,uk}u=\max\{u_1,\cdots,u_k\}u=max{u1​,⋯,uk​}, we have the following equation hold:

Pr(u≤x)=∏j=1kPr(uj≤x)=FX(x)k\text{Pr}(u\le x)=\prod_{j=1}^k\text{Pr}(u_j\le x)=F_X(x)^k Pr(u≤x)=j=1∏k​Pr(uj​≤x)=FX​(x)k

where FX(x)F_X(x)FX​(x) is the CDF of uniform distribution that is x/wx/wx/w with x∈[0,w]x\in[0,w]x∈[0,w]. Put it into Pr(u≤x)\text{Pr}(u\le x)Pr(u≤x) we have:

FU(x)=Pr(u≤x)=(xw)k,x∈[0,w]F_U(x)=\text{Pr}(u\le x)=\left(\frac{x}{w}\right)^k,x\in[0,w] FU​(x)=Pr(u≤x)=(wx​)k,x∈[0,w]

We can then sample r∼U[0,1]r\sim\mathcal{U}[0,1]r∼U[0,1] and set r=(xw)kr=\left(\frac{x}{w}\right)^kr=(wx​)k, which gives us the new w←w⋅r1/kw\gets w\cdot r^{1/k}w←w⋅r1/k. Having obtained the next selection index i+li+li+l and the new maximum value www, we can iterative this procedure until end of stream. Compared to vanilla reservoir sampling, this method does not need to read each incoming data record and rather randomly skip some of them, directly finding the next selection.

Complexity

The complexity can be divided into two parts: one for the first kkk records and the other for each upcoming record. For the first part, the complexity is fixed O(k)\mathcal{O}(k)O(k), and for the latter, the complexity is O(klog⁡(n/k))\mathcal{O}(k\log(n/k))O(klog(n/k)) (the probability of selecting xix_ixi​ is k/ik/ik/i, as stated in priority queue). Therefore, the overall complexity is O(k+klog⁡(n/k))\mathcal{O}(k+k\log(n/k))O(k+klog(n/k)).
